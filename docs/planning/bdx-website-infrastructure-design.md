> B&DX株式会社 コーポレートサイト（bdx.co.jp）の GitHub ベース管理 + CI/CD パイプライン

作成日: 2026-03-12

## 1. 全体概要

### なぜ GitHub 管理に移行したか

従来、BDX のコーポレートサイトは外部委託先（metaphase 社）が **FTP**（ファイルをサーバーに直接アップロードする仕組み）で本番サーバーに HTML を直接アップロードしていました。

この方法には以下の課題がありました。

| 課題 | 具体的な問題 |
|------|-------------|
| 履歴が残らない | 「誰が・いつ・何を変えたか」がわからない。問題が起きても原因を追えない |
| 事故リスク | 誤って本番サイトを壊しても、元に戻す手段がない |
| 確認できない | 本番に反映する前に「どう見えるか」を確認する場所がない |
| 属人化 | FTP の接続情報を知っている人しかサイトを更新できない |

**GitHub**（ソースコードの保管・履歴管理サービス）をベースにした新しい管理体制では、これらの課題をすべて解決します。

### 旧フロー vs 新フロー

```
【旧フロー】
  制作会社が HTML を編集
    → FTP で本番サーバーに直接アップロード
    → 本番サイトに即反映（確認なし）

【新フロー】
  HTML を編集して GitHub に保存
    → ステージング（確認用サイト）が自動更新
    → 表示を確認
    → 問題なければ本番デプロイ（公開）を手動実行
    → 本番サイトに反映
```

新フローでは「確認 → 公開」の2ステップを踏むため、事故を防ぎながら安全にサイトを更新できます。

## 4. ワークフロー（CI/CD）

**CI/CD**（継続的インテグレーション/継続的デリバリー）とは、コードの変更を自動的にテスト・デプロイしてくれる仕組みです。手動作業を減らし、ミスを防ぎます。

### ステージング自動デプロイ（staging.yml）

```
トリガー: main ブランチに push された時（site/** の変更がある場合のみ）
    │
    ├── 1. リポジトリのコードを取得
    │
    ├── 2. HTML 内のパスを GitHub Pages 用に自動変換
    │      例: href="/" → href="/bdx-website/"
    │      （ステージングでリンクが正しく動くようにするため）
    │
    ├── 3. GitHub Pages にアップロード
    │
    └── 4. ステージングサイトが更新される
           → https://hibiki-isogai.github.io/bdx-website/
```

| 項目 | 内容 |
|------|------|
| ファイル | `.github/workflows/staging.yml` |
| トリガー | main ブランチへの push（`site/**` 変更時のみ） |
| 実行環境 | ubuntu-latest |
| デプロイ先 | GitHub Pages |
| 所要時間 | 約 1〜2 分 |
| 手動操作 | 不要（自動実行） |

### 本番デプロイ（deploy.yml）

```
トリガー: GitHub Actions 画面で手動実行
    │
    ├── 1. 確認入力: "deploy-prod" と入力（誤操作防止）
    │      → 入力が一致しないと実行されない
    │
    ├── 2. リポジトリのコードを取得
    │
    ├── 3. FTP でさくらサーバーにアップロード
    │      接続先: tanbear66.sakura.ne.jp
    │      アップロード元: site/ フォルダ
    │      アップロード先: /home/tanbear66/www/
    │
    └── 4. 本番サイトが更新される
           → https://bdx.co.jp/
```

| 項目 | 内容 |
|------|------|
| ファイル | `.github/workflows/deploy.yml` |
| トリガー | **手動実行**（workflow_dispatch） |
| 確認入力 | `deploy-prod` と入力が必須 |
| デプロイ方法 | FTP（SamKirkland/FTP-Deploy-Action v4.3.5） |
| デプロイ先 | さくらインターネット（tanbear66.sakura.ne.jp） |
| サーバーディレクトリ | `/home/tanbear66/www/` |
| FTP 認証 | GitHub Secrets で安全に管理 |
| 安全設定 | `dangerous-clean-slate: false`（既存ファイルを全削除しない） |

### 本番デプロイの実行方法

1. GitHub で [bdx-website リポジトリ](https://github.com/Hibiki-Isogai/bdx-website) を開く
2. **Actions** タブをクリック
3. 左側から **「Deploy to Production (FTP)」** を選択
4. 右側の **「Run workflow」** ボタンをクリック
5. 確認欄に `deploy-prod` と入力
6. **「Run workflow」** をクリック

または CLI（コマンドライン）で実行:

```bash
gh workflow run deploy.yml --repo Hibiki-Isogai/bdx-website --field confirm=deploy-prod
```

## 5. 運用フロー

### 通常の更新作業

```
┌─────────────────────────────────────────────────────────┐
│ Step 1: HTML を編集                                      │
│   site/ フォルダ内のファイルを修正                          │
│   （ローカル PC でも GitHub 画面上でも OK）                  │
└──────────────────────┬──────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────┐
│ Step 2: main ブランチに push                              │
│   → ステージングが自動更新（約1〜2分）                      │
└──────────────────────┬──────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────┐
│ Step 3: ステージングで確認                                 │
│   https://hibiki-isogai.github.io/bdx-website/           │
│   → 表示が正しいか、リンクが動くか確認                      │
└──────────────────────┬──────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────┐
│ Step 4: 本番デプロイ                                      │
│   GitHub Actions → Deploy to Production → "deploy-prod"  │
│   → 本番サイト（bdx.co.jp）に反映（約1〜2分）               │
└─────────────────────────────────────────────────────────┘
```

### 緊急時の対応（サイトに問題が発生した場合）

1. GitHub の **コミット履歴**（変更の記録）から問題の変更を特定
2. **Revert**（取り消し）で変更前の状態に戻す
3. main に push → ステージングで確認 → 本番デプロイ

従来の FTP 直接アップロードでは「元に戻す」ことが難しかったですが、GitHub 管理では全ての変更履歴が残っているため、いつでも過去の状態に戻せます。

## 6. セキュリティ

### FTP 認証情報の保護

| 項目 | 管理方法 |
|------|----------|
| FTP_HOST | GitHub Secrets（暗号化して保管） |
| FTP_USER | GitHub Secrets（暗号化して保管） |
| FTP_PASSWORD | GitHub Secrets（暗号化して保管） |

- FTP の接続情報はコードに一切含まれていません
- GitHub Secrets はリポジトリの管理者のみが設定・閲覧できます
- ワークフロー実行時に自動的に読み込まれます

### 本番デプロイの安全策

| 安全策 | 説明 |
|--------|------|
| 手動トリガー | main push では本番デプロイされない。必ず手動実行が必要 |
| 確認入力 | `deploy-prod` と正確に入力しないと実行されない |
| 差分デプロイ | `dangerous-clean-slate: false` により、変更されたファイルのみアップロード。既存ファイルの全削除は行わない |

### その他

- リポジトリは public ですが、FTP 認証情報は Secrets で保護されているため漏洩リスクはありません
- GitHub Pages のステージング環境は読み取り専用（外部から改ざんされることはありません）

## 7. 注意事項・制約

### WordPress との共存

- BDX サイトの**ニュースセクション**（`/news/` 配下の一部）は **WordPress**（ブログ管理ツール）で動的に生成されています
- WordPress が管理するファイルは FTP デプロイの対象に含まれますが、`dangerous-clean-slate: false` の設定により、GitHub リポジトリにないファイルが削除されることはありません
- ただし、WordPress が生成するファイルと同名のファイルを `site/news/` に配置すると上書きされるリスクがあります

### Git リポジトリのサイズ

- 画像ファイルを大量に追加すると、リポジトリ（保管場所）のサイズが大きくなります
- 目安: リポジトリ全体で **1GB 以下** を維持することを推奨
- 大きな画像は追加前に圧縮・リサイズしてください

### GitHub Pages の制約

- GitHub Free プランでは **public リポジトリ**（誰でも閲覧可能）でのみ GitHub Pages を利用できます
- ソースコード（HTML/CSS/JS）は公開状態ですが、FTP 認証情報などの機密情報は含まれていないため問題ありません

### パス変換について

- ステージング環境では、HTML 内の絶対パス（`/` で始まるリンク）が自動的に `/bdx-website/` 付きに変換されます
- これはステージング専用の処理であり、本番デプロイ時には元のパスのままアップロードされます

## 8. 今後の課題

| 優先度 | 課題 | 説明 |
|--------|------|------|
| 高 | WordPress 共存ルール策定 | `/news/` 配下のファイル管理ルールを明確化。WordPress 管理領域と静的 HTML 管理領域を分離 |
| 中 | ブランチ戦略の導入 | 現在は main に直接 push していますが、**ブランチ**（作業用の分岐）を作成 → **プルリクエスト**（変更の確認依頼）→ レビュー → main にマージ、というフローに移行すると、複数人での作業がより安全になります |
| 中 | 画像最適化 | 画像の自動圧縮・WebP 変換パイプラインの導入。サイト表示速度の向上とリポジトリサイズの削減 |
| 低 | ステージング URL の短縮 | 現在の GitHub Pages URL は長いため、カスタムドメイン（例: `staging.bdx.co.jp`）の設定を検討 |
| 低 | HTML バリデーション | push 時に HTML の構文チェックを自動実行し、壊れた HTML が本番に公開されることを防止 |

## 付録: 用語集

| 用語 | 説明 |
|------|------|
| **GitHub** | ソースコードの保管・履歴管理・共同作業のためのサービス |
| **リポジトリ** | プロジェクトの保管場所。ファイルと変更履歴がまとめて管理される |
| **main ブランチ** | リポジトリのメインの保管場所。ここにあるコードが「正」 |
| **push** | ローカル PC での変更をサーバー（GitHub）に送ること |
| **GitHub Pages** | GitHub が無料で提供する Web サイト公開機能 |
| **GitHub Actions** | GitHub 上でプログラムを自動実行する仕組み（CI/CD） |
| **CI/CD** | コードの変更を自動的にテスト・デプロイしてくれる仕組み |
| **FTP** | ファイルをサーバーに転送する通信方式 |
| **GitHub Secrets** | パスワードなどの機密情報を暗号化して GitHub に保管する機能 |
| **workflow_dispatch** | GitHub Actions を手動で実行するトリガー |
| **ステージング** | 本番に反映する前の確認用環境 |
| **デプロイ** | 作ったものをサーバーに配置して使えるようにすること |
| **コミット** | ファイルの変更を記録すること。いつでも過去の状態に戻せる |
| **Revert** | 過去のコミットを取り消して、変更前の状態に戻すこと |
